# 3-Agent 사용 가이드: 언제 어떤 방식을 사용할까?

## 두 가지 접근 방식

### 방식 1: 직접 지시 (기존 방식)
```
"A 기능을 확인하고 수정해줘"
```

### 방식 2: 3-Agent 분산 처리 (새로운 방식)
```
Agent 1: 현재 이슈를 분석해줘
Agent 2: 분석된 내용으로 설계해줘
Agent 3: 구현된 코드를 테스트해줘
```

---

## 비교 분석

### 방식 1: 직접 지시

**장점:**
- ✅ 빠른 피드백 (한 번에 처리)
- ✅ 간단한 작업에 적합
- ✅ 즉시 결과 확인 가능
- ✅ 소규모 수정에 효율적

**단점:**
- ❌ 복잡한 작업 시 품질 저하 가능
- ❌ 설계 없이 바로 구현 → 리팩토링 필요
- ❌ 테스트 누락 가능성

**적합한 상황:**
- 버그 수정 (명확한 문제)
- 작은 리팩토링
- 간단한 기능 추가
- 긴급한 수정

**예시:**
```
"reverse.comparison 규칙에서 '·' 같은 특수 문자를 필터링해줘"
→ 바로 코드 수정 가능
```

---

### 방식 2: 3-Agent 분산 처리

**장점:**
- ✅ 체계적인 접근 (분석 → 설계 → 구현)
- ✅ 높은 품질 보장
- ✅ 문서화 자동화
- ✅ 복잡한 작업에 적합
- ✅ 재사용 가능한 설계 문서

**단점:**
- ❌ 시간이 더 걸림 (3단계)
- ❌ 간단한 작업에는 오버헤드
- ❌ 에이전트 간 커뮤니케이션 필요

**적합한 상황:**
- 새로운 기능 개발
- 대규모 리팩토링
- 아키텍처 변경
- 복잡한 문제 해결
- 장기 프로젝트

**예시:**
```
Agent 1: "새로운 diff 규칙 추가 요구사항 분석"
Agent 2: "규칙 인터페이스 설계"
Agent 3: "구현 및 테스트"
```

---

## 추천: 하이브리드 접근법

### 상황별 선택 가이드

#### 🟢 간단한 작업 → 방식 1 (직접 지시)

**예시:**
```
✅ "버튼 텍스트 오타 수정해줘"
✅ "console.log 제거해줘"
✅ "타입 에러 수정해줘"
✅ "간단한 필터링 로직 추가해줘"
```

**실행:**
- 한 번에 처리
- 빠른 피드백

---

#### 🟡 중간 복잡도 → 방식 1 또는 2 선택

**예시:**
```
🤔 "노이즈 필터링 강화" 
   → 간단하면 방식 1, 복잡하면 방식 2

🤔 "새로운 유틸리티 함수 추가"
   → 간단하면 방식 1, 여러 함수면 방식 2
```

**판단 기준:**
- 파일 수: 1-2개 → 방식 1
- 파일 수: 3개 이상 → 방식 2
- 테스트 필요: 간단 → 방식 1
- 테스트 필요: 복잡 → 방식 2

---

#### 🔴 복잡한 작업 → 방식 2 (3-Agent)

**예시:**
```
✅ "새로운 diff 규칙 엔진 개발"
✅ "Phase 3 지원 추가"
✅ "성능 최적화 프로젝트"
✅ "대규모 리팩토링"
```

**실행:**
- Agent 1: 요구사항 분석
- Agent 2: 설계 문서 작성
- Agent 3: 구현 및 테스트

---

## 실제 사용 예시

### 예시 1: 간단한 버그 수정 (방식 1)

**지시:**
```
"reverse.comparison 규칙에서 '·' 같은 특수 문자를 필터링해줘.
rules.ts 파일의 reverseComparisonRule 함수를 수정하면 돼."
```

**결과:** 바로 수정 완료

---

### 예시 2: 중간 복잡도 (방식 1 또는 2 선택)

**지시 (방식 1 선택 시):**
```
"노이즈 필터링을 강화해줘. 
특수 문자, 숫자만 있는 텍스트, 해상도 라벨, 메타데이터 라벨, 상태 라벨을 필터링해야 해."
```

**지시 (방식 2 선택 시):**
```
Agent 1: "노이즈 필터링 강화 요구사항 분석해줘"
→ docs/agents/agent-1-requirements/analysis-results/noise-filtering.md

Agent 2: "분석 결과를 읽고 필터링 로직 설계해줘"
→ docs/agents/agent-2-design/design-docs/noise-filtering-design.md

Agent 3: "설계를 읽고 구현하고 테스트해줘"
→ 구현 코드 + 테스트 결과
```

---

### 예시 3: 복잡한 기능 개발 (방식 2 필수)

**지시:**
```
Agent 1: "새로운 diff 규칙 'color.diff' 추가 요구사항 분석해줘"
→ 텍스트 색상 비교 규칙이 필요한 이유, 요구사항 도출

Agent 2: "분석 결과를 읽고 colorDiffRule 인터페이스 설계해줘"
→ 규칙 인터페이스, 매칭 로직 설계

Agent 3: "설계를 읽고 구현하고 테스트해줘"
→ rules.ts에 추가, 테스트 코드 작성, 검증
```

## 실전 팁

### 1. 처음에는 방식 1로 시작

**이유:**
- 빠르게 결과 확인 가능
- 문제가 복잡해지면 방식 2로 전환

**예시:**
```
1차: "버그 수정해줘" (방식 1)
→ 복잡한 문제 발견
2차: Agent 1에게 "이 버그 분석해줘" (방식 2로 전환)
```

---

### 2. 방식 2는 문서화가 자동화됨

**장점:**
- 분석 문서 자동 생성
- 설계 문서 자동 생성
- 테스트 결과 자동 생성
- 나중에 참고 가능

---

### 3. 병렬 처리 가능

**예시:**
```
Agent 1-1: 작업 A 분석
Agent 1-2: 작업 B 분석 (동시 진행 가능)

Agent 2-1: 작업 A 설계 (Agent 1-1 완료 후)
Agent 2-2: 작업 B 설계 (Agent 1-2 완료 후)
```

---

## 결론 및 추천

### 기본 원칙

1. **간단한 작업**: 방식 1 (직접 지시)
2. **복잡한 작업**: 방식 2 (3-Agent)
3. **불확실할 때**: 방식 1로 시작 → 필요시 방식 2로 전환

### 추천 워크플로우

```
작업 시작
    ↓
간단한가? → Yes → 방식 1 (직접 지시)
    ↓ No
복잡한가? → Yes → 방식 2 (3-Agent)
    ↓ No
중간 복잡도 → 방식 1로 시작, 필요시 방식 2로 전환
```

### 최종 권장사항

**대부분의 경우 방식 1을 사용하고,**
**복잡하거나 중요한 작업만 방식 2를 사용하세요.**

이렇게 하면:
- ✅ 빠른 작업은 빠르게 처리
- ✅ 중요한 작업은 체계적으로 처리
- ✅ 최적의 효율성 달성
